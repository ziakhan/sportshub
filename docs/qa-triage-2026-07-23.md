## §1 · P0 — Root-caused bugs, ready to fix

### QA-001 · Trainer has no way back to their workspace — BUG
**Found:** trainer on the public site has no route back to dashboard except the
notifications bell.
**Root cause:** `Trainer` role exists (`prisma/schema.prisma:248`) and is granted
on trainer creation (`api/trainers/route.ts:87-92`), but `OPERATOR_ROLES` in
`apps/web/src/lib/queries/nav.ts:32-42` was never updated — so `isOperator` is
false and the public header's Manage door (`(public)/layout.tsx:111`) never
renders for trainers.
**Fix:** add `"Trainer"` to `OPERATOR_ROLES`. One line.

### QA-002 · No dashboard door at all on mobile widths — BUG
**Found:** (generalizes QA-001) on a phone there is no way into the workspace
for *any* operator.
**Root cause:** Manage/My Hub button is `hidden … sm:inline-flex`
(`(public)/layout.tsx:114`) — gone below 640px — and the mobile pill row
(`components/public/section-pills.tsx:22-61`) has no Dashboard/Manage entry.
**Fix:** add a Manage/My Hub pill (or keep the button visible at all widths).

### QA-003 · Uploaded profile photo never shows in the header — BUG
**Found:** uploaded a profile photo; top-corner avatar never updates.
**Root cause:** `User.avatarUrl` is saved (`schema.prisma:25`) but never
threaded to the header: `dashboard/user-menu.tsx:43-45` and
`components/nav/account-menu.tsx:104` render initials only — neither accepts an
image prop; layouts compute initials and never pass `avatarUrl`.
**Fix:** pass `avatarUrl` from both layouts; render `<img>` with initials
fallback in both menus.

### QA-004 · Ref-assign widget hangs on "Loading…" — BUG (two-part)
**Found:** in the scoring hub, the assign-referee search never resolves.
**Root cause (a):** `components/scoring/game-referee-control.tsx:34-41` —
`loadPool` does `fetch(...).catch(() => null)` then `if (!res?.ok) return`,
never setting `pool` or `error`; `pool` stays `null` → permanent "Loading…"
(`:112`). Any 403/network failure = infinite spinner.
**Root cause (b):** the widget renders for viewers who can't use it. Authz is
`canScoreGame` (`lib/scoring/authz.ts:46-80`): scorekeeper, league owner,
club/team staff — **not referees**. A referee viewing their own game sees the
control, gets 403, hits (a). (Corrects the tester's read: refs can *not*
actually assign refs — the UI just pretends they can, then hangs.)
**Fix:** surface fetch errors + hide the control when the viewer lacks scoring
authz.

### QA-005 · Venue search finds almost nothing — BUG (env) + BUG (UX)
**Found:** searched multiple real gyms previously used by leagues; only a
handful of venues ever appear.
**Root cause:** the selector has two sources. (1) "Existing Venues" = the
platform `Venue` directory (substring match, `api/venues/route.ts:120-139`) —
which contains only ~4 seeded venues (`scripts/seed-nph-demo.ts:112-117`) plus
manual adds; free-text-era program locations never created Venue rows. (2)
"Add New from Google Maps" = Places autocomplete that **fails silently** when
`NEXT_PUBLIC_GOOGLE_PLACES_API_KEY` is missing/invalid — a `console.warn` only
(`components/venue-selector.tsx:52-57`). Symptom exactly matches the Google
layer being dead on the box.
**Fix:** (1) verify the Places key is present **at build time** on the box
(`NEXT_PUBLIC_` vars are inlined at build; setting it post-build does nothing)
and unrestricted for the live domain — check browser console for
`RefererNotAllowed…`/`ApiNotActivated…`/`BillingNotEnabled…`; (2) make the
selector visibly say "map search unavailable" instead of failing silently.

### QA-006 · Field-level validation errors are swallowed — BUG
**Found:** create house league/camp → generic red "Validation error" banner,
no clue which field.
**Root cause:** APIs return `{ error: "Validation error", details }`
(`api/camps/route.ts:92-96`, `api/house-leagues/route.ts:93-97`) but the create
pages discard `details` (`camps/create/page.tsx:78-83`,
`house-leagues/create/page.tsx:93-98`).
**Fix:** surface zod `details` per-field (or map into the banner).

### QA-007 · Program capacity check races (overfill possible) — BUG
**Found by code review while verifying tester's capacity concerns.**
**Root cause:** all four signup routes read the active-signup count and check
"full" **before** the `$transaction`, not inside it (camps `signup/route.ts:64`
vs tx `:143`; house-leagues `:62`/`:126`; tryouts `:72`/`:154`; training
`:63`/`:128`). Two concurrent signups can both pass.
**Fix:** re-check count inside the transaction (or unique-constraint style
guard).

### QA-008 · ICS calendar feed omits refereeing games — BUG
**Found:** ref's subscribable calendar lacks their assignments (in-app calendar
now has them — see QA-102).
**Root cause:** `api/calendar/[token]/route.ts:70-75` includes only games where
home/away ∈ the user's member teams; officiating assignments
(`UserRole role=Referee gameId≠null`) are never queried.
**Fix:** union in refereeing games (mirror `lib/calendar/my-calendar.ts:87-91`).

### QA-009 · Score fold treats `made: null` as a made shot — BUG (latent)
**Found by code review while chasing the tester's "+4 on miss+rebound" report.**
**Root cause:** `lib/scoring/fold.ts:254` `const made = e.made !== false` —
a score event with `made === null` counts +2/+3 in the fold while the stored
`points` column says 0 (`api/games/[id]/events/route.ts:186-193`) →
score-vs-boxscore divergence path. (The reported miss+rebound→+4 itself is NOT
reproducible in current code: rebounds carry no points; likely a double-tap or
old build — see QA-301.)
**Fix:** treat only `made === true` (or backfill/forbid null) — pick one
convention across fold + events.

## §2 · P1 — Decisions needed (works as coded today)

### QA-101 · Referee pay: who sets the rate? — DECISION
Tester: "shouldn't it be up to the league's discretion?" Current model is the
inverse of real-world convention: `RefereeProfile.standardFee` is set by the
referee (onboarding + profile), leagues cannot set per-game/per-level rates
(`LeagueReferee` and `RefereeSessionRequest` carry no fee fields,
`schema.prisma:1510-1570`), and the fee is **display-only** — `REFEREE_FEE`
enum value has zero code references; no obligation/payout is ever created on
assign or finalize. Real-world norm: league/assignor publishes a rate schedule
(tiered by age level and role), refs accept at that rate.
**Decide:** league-set rate schedule (recommended), and whether/when to wire
actual payout obligations.

### QA-102 · Ref scheduling: booking unit + no-taker fallback — DECISION
Tester: "refs should book by the weekend… what happens if refs don't book?"
Current: `RefereeSessionRequest` is strictly per session-day, first-accept-wins;
an unaccepted broadcast sits PENDING forever — no escalation, no auto-assign,
no reminder (`api/leagues/[id]/referee-requests/route.ts:82-158`,
`referee-requests/[id]/route.ts:98-148`). Game-day manual assign by
staff/scorekeeper is the only net.
**Decide:** weekend/multi-day bundled offers? escalation policy (re-broadcast →
notify league → flag game unstaffed)?

### QA-103 · Offer/roster cap — DECISION then build
Tester (twice): 18 offers for a 12-man roster; accepts keep working after the
roster is full. Verified: **no roster-cap field exists anywhere** (Team,
OfferTemplate, Offer — `schema.prisma:397-452,1022-1081`) and `acceptOffer`
upserts TeamPlayer with zero capacity check
(`lib/offers/respond-to-offer.ts:148`). Only a *minimum* exists
(`MIN_ROSTER_SIZE = 5`, `api/teams/[id]/finalize/route.ts:11`).
Over-inviting is fine (declines expected); accepting past capacity is the bug.
**Decide:** where the cap lives — Team setting, offer template, or league rule —
then enforce at accept (and badge remaining spots on the offers screen).

### QA-104 · Tryouts force draft-first — DECISION
`api/tryouts/route.ts:71` hardcodes `isPublished: false`; publish is a separate
POST (`[id]/publish/route.ts`). Add "Save & publish" on create?

### QA-105 · Practice times are public once announced — DECISION
`(public)/team/[id]/page.tsx:53-66` shows practice schedule to everyone after
staff announce it. Tester asks whether the public needs this. Members-only
toggle?

### QA-106 · Minors (13+) can become ClubOwner/Referee/LeagueOwner — DECISION
Under-13 is COPPA-blocked (`api/onboarding/route.ts:80-92`), but 13+ accounts
get the full role selector (`onboarding/role-selector.tsx:6-45`) and
`/clubs/create` unconditionally (`account-menu.tsx:161-164`). Tester: block
add-player/club/league/referee for minors; require adult for payments.
**Decide:** age-gate operator roles at onboarding + route guards.

### QA-107 · Division "tier" unexplained — DECISION
Tier select still exposed (`divisions-tab.tsx:237-245`, Tier 1 Top/2/3) with no
explanation. Explain (tooltip/copy) or hide behind an "advanced" toggle.
(Max-teams input is fixed — labeled, "blank = unlimited", `:123-125,:247-255`.)

### QA-108 · Recap headline inconsistency — DECISION
`recap-service.ts:133-135`: Claude authors free-form headlines when
`ANTHROPIC_API_KEY` is set; template recaps use the fixed
"{winner} {verb} {loser} {score}" pattern (`recap.ts:156-159`). Mixed feeds look
inconsistent (tester noticed). Constrain the Claude prompt to the house title
format, or accept variety.
